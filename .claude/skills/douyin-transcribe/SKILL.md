# douyin-transcribe — 抖音视频转写（豆包办公任务 + 剪贴板取回）

把抖音视频转成完整文字稿的一条**固定通道**：
调试 Chrome → 豆包固定入口 → 发转写提示词 → 等"办公任务"跑完 → **点豆包「复制」按钮、读系统剪贴板**拿回干净全文 → **正文到手立即关浏览器** → 再按用户诉求处理。

**为什么这么设计**：豆包能直接对抖音链接做语音转写（实测抖音很顺、基本不触发登录/验证）；取结果**走它自带的「复制」按钮 + 系统剪贴板**——比读 DOM `innerText` 干净（后者能拿到全文但会夹带"办公任务/本地电脑/技能"等界面噪声，还得自己清洗）。

## 关键约定（先记牢）

- **登录态不写进本 skill**：豆包登录由持久 profile 自动带上。**绝不**在 skill 里写账号密码、绝不抓 cookie。
- **只认抖音**：从用户文案里抠出纯链接 `https://v.douyin.com/xxx/`（去掉分享口令、标题、`?` 后的多余参数）。
- **完成判定只信 DOM / 快照，不信截图或可视区**：长文转写时视图常停在顶部、底部动作栏在屏幕外，凭"看见没看见"会把"已完成"误判成"没完成"而卡死。判完成、取内容都要靠 DOM/快照（它们与滚动位置无关，屏幕外的元素照样能查到）。
- **点按钮认身份、不认坐标**：`take_snapshot` 拿到目标按钮的 **uid**，`click(uid)` 会自动把它滚进视图再点——**哪怕在屏幕外也点得到**（已实测）。绝不用固定坐标找按钮，也无需手动滚动。
- **脚本一律绝对路径**：把 `<仓库根>`（见 CLAUDE.md「路径占位」说明）展开为仓库克隆的实际绝对根路径再调用，例如 `<仓库根>\scripts\…`；用相对路径会被当成相对 skill 目录解析而"找不到文件"。

## 第 1 步：确保调试 Chrome 已起（端口 9222）

复用 web-scrape 的浏览器层。先探测，**已在跑就别重开**（复用登录态、避免多开）：

```bash
curl -s -m 2 http://127.0.0.1:9222/json/version
```

- 有 JSON 返回（含 `webSocketDebuggerUrl`）→ 已就绪，进第 2 步。
- 连不上 → 用 VBS 拉起（无 CMD 窗口、不杀主 Chrome、专用持久 profile），**必须绝对路径**：

```bash
wscript "<仓库根>\scripts\launch-scrape-chrome.vbs"
```

然后**轮询** `curl ...:9222/json/version` 直到有返回（约 1–6 秒）再继续。

## 第 2 步：打开豆包固定入口

用 `chrome-devtools` MCP 的 `navigate_page`（`type:"url"`）打开这个固定地址：

```
https://www.doubao.com/?channel=cainiaoplus&source=hw_db_cainiaoplus&type=ziy&theme=bianc
```

已登录会自动落到 `/chat`。可 `list_pages` 确认当前页。

## 第 3 步：填提示词并发送

1. `take_snapshot` 找到底部输入框（multiline textbox）。
2. `fill` 写入提示词（把 `<URL>` 换成抠出来的纯抖音链接）：
   ```
   帮我转写这个抖音视频的完整文字稿：<URL>，要求：完整语音转写，输出纯文字，不要字幕。
   ```
3. 填好后输入框旁会**新出现一个发送按钮**，`click` 它发送。
4. **坑**：点发送后页面有延迟、且会跳到新对话 URL（`/chat/...`）。别用点击后立刻的那张快照判断——**重新 `take_snapshot` 确认已提交**（出现用户气泡 + 豆包开始"办公任务"）。

## 第 4 步：等"办公任务"完成 —— 正常 / 意外 两条分支

豆包进入"办公任务"自动执行（启动→规划→转写），这是**异步慢任务**。轮询监测，**只用 DOM/快照判断、别靠截图可视区**（长文时动作栏在屏幕外，看截图会卡死）。在这里**分两条路**：

### ① 正常 → 取结果
用 `evaluate_script`（或 `take_snapshot`）同时确认三点，全满足才算完成：

- 助手回复下方动作栏出现**带 `aria-label="朗读"` 的按钮**——它只在该条消息生成完毕后才有，且在 DOM/快照里即使滚出屏幕也找得到，所以最可靠；
- `main.innerText` 长度**连续两次轮询不再增长**（确认不是还在流式输出）；
- 页面无"停止 / 生成中 / 思考中"等字样。

满足 → 进第 5 步；否则继续轮询。探测脚本参考：

```
() => {
  const t = (document.querySelector('main') || document.body).innerText || '';
  const hasRead = [...document.querySelectorAll('button')].some(b => (b.getAttribute('aria-label')||'') === '朗读');
  const generating = /停止|生成中|正在(生成|思考|转写)|思考中/.test(t);
  return { len: t.length, hasRead, generating };   // len 连续两次相同 && hasRead && !generating ⇒ 完成
}
```

### ② 意外 → 触发兜底：立即中断 + 发飞书
只要出现下列任一**非正常态**：

- 等待**超时**仍无文字稿（已给足时间、数分钟无进展）；
- 页面文本出现 **登录 / 验证 / "请求用户协助" / "去操作"** 等字样（抖音极少见，但兜底必须在）；
- 页面出现**明显报错 / 失败**提示（服务繁忙、生成失败、网络错误等——豆包官方服务偶发不稳定，多次失败常是这种）。

就**立即中断**：不重试、不继续、**绝不编造**文字稿。随后经飞书桥接**主动推一条消息**讲清原因（这条通道独立可跑、不要求常驻桥接在线）：

```bash
node "<仓库根>\bridge\send-reminder.js" "<中断原因，可中文/markdown>" "抖音转写中断"
```

例：`node "<仓库根>\bridge\send-reminder.js" "抖音转写已中断：页面要求登录/验证，需人工处理。" "抖音转写中断"`

推送后本次结束。**意外中断时浏览器现场先留着**（便于排查/人工处理），不强制关（见第 7 步）。

## 第 5 步：取结果 —— 用「复制」按钮 + 读剪贴板

拿到**干净完整全文**的取法（别直接读 `innerText` 抓——那会夹带界面噪声还得清洗）：

1. `take_snapshot` 定位该条回复的动作栏：找到带 `aria-label="朗读"` 的按钮，**「复制」就是它紧挨左边那个无文字标签的按钮**——**按身份认（朗读左邻），不认位置/坐标**。记下复制按钮的 uid。
2. 给剪贴板写哨兵值占位（用于验证复制是否真生效）：
   ```bash
   powershell -NoProfile -Command "Set-Clipboard -Value '__CLIP_SENTINEL__'"
   ```
3. `click(<复制按钮 uid>)`：`click` 会自动把它滚进视图，**哪怕在屏幕外也点得到**，无需手动滚动、无需坐标。（必须是真实点击=用户手势，剪贴板写入才生效。）
4. 读回并校验：
   ```bash
   powershell -NoProfile -Command "$c=Get-Clipboard -Raw; if($c -and $c -ne '__CLIP_SENTINEL__'){$c}else{'COPY_FAILED'}"
   ```
5. 读到的就是规整文字稿（自带分段、无界面噪声）。若得 `COPY_FAILED` → 复制按钮没认对：重新 `take_snapshot`，把「朗读」左邻那个按钮的 uid 再点一次；必要时挨个试动作栏按钮，直到剪贴板 ≠ 哨兵。仍不行 → 按第 4 步②兜底中断并通知。
6. **剪贴板已取到正文（≠ 哨兵）= 转写确凿完成的最硬信号——此刻立即关浏览器**（执行第 7 步那条 `close-scrape-chrome.ps1`），关掉之后再回头做第 6 步的处理/回复。别把关闭拖到"回复用户之后"，那一步最容易漏。正文已完整在手、后续是纯文本处理、不需再回看页面，所以在这里关最安全。

## 第 6 步：按用户诉求处理全文

拿到全文后，**按用户当初那句话**决定去向（不擅自决定、不顺手揽活）：

- 只要文字稿 → 直接完整回给用户。
- 要"写入知识库" → 转交 **knowledge-write**（原子化 + 四段式 + **二次确认**后写 `knowledge/`，再跑 kg 索引）。本 skill **不直接写** `knowledge/`。
- 临时分析/草稿 → 放 `workspace/`。
- 拿不准就问用户。

## 第 7 步：关浏览器（触发点已前移到第 5 步取到正文后）

**关闭时机 = 第 5 步剪贴板拿到正文那一刻**，别拖到"回复用户之后"再关（那是最常见的"忘关"原因）。关闭命令：

```bash
powershell -NoProfile -ExecutionPolicy Bypass -File "<仓库根>\scripts\close-scrape-chrome.ps1"
```

它只关"调试实例"（按 9222 / 专用 profile 匹配），**不碰**你的日常 Chrome。

- **判断该关只认一个硬信号：剪贴板已取到正文（≠ 哨兵）**。一旦成立就立刻关，别用"视图还停在顶部、没看见底部动作栏"之类可视错觉去怀疑"是不是没完成"。
- **兜底**：若不慎拖到了回复用户那一步才发现浏览器还开着，立即补关一次（正常路径不该走到这）。
- 例外（保持不关）：**第 4 步②意外中断**时留现场便于排查；用户明确说要留着继续看，也别关。

## 边界

- 只在本仓库范围内活动；脚本一律绝对路径 `<仓库根>\scripts\…`。
- **仅抖音**；B站/YouTube 等其它平台不走本 skill。
- 登录态靠持久 profile，不写账号密码进 skill，不抓 cookie。
- 取结果走「复制」按钮 + 剪贴板；入库一律转交 knowledge-write 走二次确认。
- 真要操作浏览器就走这条通道，别用 `WebFetch`/`WebSearch` 代替——它们到不了登录会话、也拿不到语音内容。
